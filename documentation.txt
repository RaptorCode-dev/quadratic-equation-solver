Решение квадратных уравнений с длинной арифметикой


ЧТО ДЕЛАЕТ ПРОГРАММА

Программа решает квадратное уравнение a·x² + b·x + c = 0 с очень высокой точностью — до 10000 знаков после запятой.
Обычный float в Python даёт всего 15-16 знаков, а встроенный модуль Decimal хоть и работает с любой точностью, но
медленный на больших числах. Поэтому я написал свою длинную арифметику с нуля и реализовал на ней все нужные операции:
сложение, вычитание, умножение, деление и квадратный корень.

Главная идея — представить число в виде списка «блоков» по 5 цифр и работать с этими блоками как с цифрами обычной
арифметики, только не в десятичной системе, а в системе по основанию 10⁵ = 100000. Это даёт компромисс между скоростью
(мало блоков на 10000 знаков — всего 2000 штук) и простотой реализации (каждый блок умещается в обычный int).


СТРУКТУРА ПРОЕКТА

class_1.py        — структура BigFloat и базовые операции

МОДУЛЬ class_1.py — ОСНОВА ВСЕГО

Класс BigFloat

Это сердце программы. Каждое число хранится тремя полями:

  - sign — знак: 1 или -1.
  - mantissa — список блоков. Например, число 123456789 хранится как [56789, 1234].
    Младшие блоки идут первыми, старшие — в конце. Это удобно потому что когда число
    становится длиннее, новые блоки просто дописываются в конец списка через append.
  - exp — экспонента (показатель степени десятки). Само число равно mantissa · 10^exp.

Например, число 0.00123456789 можно записать как:

    BigFloat(exp=-11, mantissa=[56789, 1234], sign=1)

Потому что 123456789 · 10⁻¹¹ = 0.00123456789.

Я использую __slots__ = ('exp', 'mantissa', 'sign') чтобы Python не создавал словарь
для каждого объекта — это экономит память и ускоряет доступ к полям.


МОДУЛЬ AddandSub.py — СЛОЖЕНИЕ И ВЫЧИТАНИЕ

Главная идея

Сложение и вычитание — школьное «в столбик», только цифры теперь не от 0 до 9, а от 0
до 99999. Перенос ловится когда блок переваливает за 10⁵.

Функции add и sub

add(a, b) работает так:

1. Делает копии чисел чтобы не портить оригиналы.
2. Выравнивает экспоненты — если у одного числа exp = 5, а у другого exp = 2, нужно
   сдвинуть младшее так, чтобы оба числа были «в одной системе координат».
3. Если знаки совпадают — складывает мантиссы.
4. Если знаки разные — вычитает (по модулю), и знак результата равен знаку большего
   по модулю числа.
5. Нормализует результат.

sub(a, b) я сделал коротким — это просто add(a, -b). На копии меняем знак второго числа
и делегируем работу в add. Это убрало дублирование кода.


МОДУЛЬ mul.py — УМНОЖЕНИЕ ЧЕРЕЗ FFT

Почему не в столбик

Умножение в столбик имеет сложность O(n²). При длине числа 2000 блоков это 4 миллиона
операций — слишком медленно. Поэтому я использую алгоритм через быстрое преобразование
Фурье (FFT) со сложностью O(n·log n) — для тех же 2000 блоков это всего ~22000 операций.

Идея

Каждое число можно рассматривать как полином: число [a₀, a₁, a₂] это полином
A(x) = a₀ + a₁·x + a₂·x² при x = 100000. Умножение чисел = умножение полиномов =
свёртка их коэффициентов.

Полином можно задать не коэффициентами, а значениями в нескольких точках. Тогда
умножение полиномов — это просто поточечное умножение значений в одних и тех же точках.
FFT позволяет быстро переводить туда и обратно.

Алгоритм:

1. Берём мантиссы a и b, дополняем нулями до размера степени двойки.
2. Применяем FFT — переводим в «частотное» представление.
3. Поточечно перемножаем (это O(n)).
4. Применяем обратное FFT — получаем коэффициенты результата.
5. Делаем переносы между блоками (потому что после свёртки блоки могут быть больше 10⁵).


МОДУЛЬ div.py — ДЕЛЕНИЕ МЕТОДОМ НЬЮТОНА

Идея

Делить «в столбик» для длинной арифметики неудобно — это сложно реализовать и медленно.
Вместо этого я нахожу 1/b итерационно (это и есть метод Ньютона), а потом просто
умножаю на a.

Формула Ньютона для нахождения 1/x:

    y_{k+1} = y_k · (2 - x · y_k)

Эта формула удваивает число верных цифр на каждой итерации. Если первое приближение даёт
~15 верных цифр, то за 11 итераций получим 15 · 2¹¹ = ~30000 цифр — с большим запасом
для нашей точности 10000.


МОДУЛЬ sqrt.py — КВАДРАТНЫЙ КОРЕНЬ

Идея

Тут двойной фокус: для нахождения √n я сначала ищу 1/√n, и потом домножаю на n. Зачем?
Формула Ньютона для 1/√n не содержит деления:

    y_{k+1} = y_k · (3 - n · y_k²) / 2

А для прямого √n пришлось бы делить на y_k, а деление — это само по себе целая отдельная
операция. Так что искать обратный корень дешевле, и финальное √n = n · (1/√n) стоит
всего одно умножение.


МОДУЛИ ВВОДА — interpretater.py И input.py

Зачем отдельный интерпретатор

Я использовал паттерн Interpreter: для каждой части синтаксиса (число, экспонента, точка,
мантисса) свой класс с методом interpret(). Это сильно упрощает тестирование: можно
отдельно проверить парсинг экспоненты, отдельно — мантиссы.

Структура

ExpressionParser принимает строку из 3 чисел через пробел и для каждого вызывает
NumberParser. Тот, в свою очередь:

1. Через parse_sign отщепляет + или -.
2. Через split_exp отделяет экспоненту (часть после e или E).
3. ExponentParser разбирает экспоненту в обычное число.
4. DotParser находит положение точки и вычисляет смещение для exp.
5. После удаления точки оставшаяся строка цифр идёт в MantissaParser, который режет её
   на блоки по 5 цифр (справа налево) через DigitChunkParser.

Если что-то идёт не так — вызывается fail(msg), который печатает сообщение и завершает
программу.

input.py — это просто тонкая обёртка: читает строку через input() и прогоняет её через
ExpressionParser. Возвращает объект Coefs с тремя коэффициентами.


МОДУЛЬ solve.py — САМО РЕШЕНИЕ

Главная функция

    def solve(coefs):
        if is_linear(coefs):
            return linear_solution(b, c)
        return quadratic_solution(a, b, c)

Если a == 0, уравнение линейное b·x + c = 0. В этом случае:
  - если b == 0 и c == 0 — бесконечно решений (0 = 0),
  - если b == 0 и c ≠ 0 — нет решений (5 ≠ 0),
  - иначе один корень x = -c / b.


МОДУЛЬ output.py — ВЫВОД И ПРОВЕРКА

Этот модуль не просто печатает — он ещё проверяет мой ответ через Decimal. Идея:
посчитать те же корни через стандартный модуль Decimal (это эталон) и сравнить.


ТЕСТИРОВАНИЕ

В файле testr.py — общий тест всех операций и решения целиком. Структура:

  1. Замеряется время самой операции через perf_counter().
  2. После замера считается эталон через Decimal и сравнивается. Это важно: проверка не
     входит во время бенчмарка.
  3. Печатается строка вида "0.022s умножение OK".

Для квадратки сравнение чуть хитрее — учитываю что порядок корней может быть любым и
что у комплексных корней знак мнимой части может быть в любом из x1/x2.

Типичный результат прогона:

    0.000599s  сложение    OK
    0.000515s  вычитание   OK
    0.014477s  умножение   OK
    0.125221s  деление     OK
    0.205782s  корень      OK
    0.514424s  квадратка   OK
